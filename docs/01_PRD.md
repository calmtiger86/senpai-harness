# 01. Product Requirements Document

## 제품명

Senpai Harness

## 제품 유형

Claude Code CLI에서 동작하는 Obsidian 기반 비개발자용 Guided Auto-Drive AI Harness.

## 대상 사용자

주 사용자는 비개발자입니다.

대표 사용자:

- 코딩 문법을 모르는 사람
- Claude Code CLI, Codex, 터미널을 처음 쓰는 사람
- 바이브코딩으로 앱, 챗봇, 자동화, 콘텐츠 시스템을 만들고 싶은 사람
- AI가 만들어주는 결과를 따라가기 어려워하는 사람
- 프로젝트가 커지면 맥락을 잃고 포기하는 사람

## 사용자 문제

비개발자는 “무엇을 모르는지 모르는 상태”로 바이브코딩을 시작합니다.

이 상태에서 AI가 곧바로 구현하면 다음 문제가 생깁니다.

- 사용자는 숨은 결정을 인지하지 못합니다.
- AI가 임의로 기술적 결정을 내립니다.
- 사용자는 프로젝트 구조가 왜 커지는지 모릅니다.
- 오류가 생기면 원인과 해결 과정이 흩어집니다.
- 다음 세션에서 맥락을 다시 설명해야 합니다.
- 완료 기준을 알 수 없어 결과를 신뢰하기 어렵습니다.
- 후반으로 갈수록 프로젝트를 따라가지 못하고 포기합니다.

## 제품 목표

Senpai Harness의 목표는 비개발자가 AI와 함께 프로젝트를 끝까지 따라갈 수 있게 만드는 것입니다.

구체 목표:

1. 사용자가 명령어를 외우지 않아도 작업 흐름을 자동으로 리드합니다.
2. 새 기능이나 큰 변경 전, 숨은 결정을 자동으로 드러냅니다.
3. 사용자가 이해할 수 있는 회의 카드와 선택지를 제공합니다.
4. 사용자 승인 전에는 제품 코드를 수정하지 않습니다.
5. 승인된 계획 안에서만 구현합니다.
6. 완료 전에는 검증 증거를 요구합니다.
7. 결정, 오류, 작업 기억, 검증 결과를 Obsidian에 저장합니다.
8. 작업 특성에 맞게 병렬 에이전트와 모델 티어를 자동 라우팅합니다.

## 핵심 가치 제안

> 명령어, 에이전트, 모델, 개발 순서를 몰라도 됩니다. Senpai Harness가 먼저 상황을 읽고, 회의를 열고, 숨은 결정을 드러내고, 사용자가 이해한 뒤에만 구현으로 넘어갑니다.

## 주요 사용 시나리오

### 시나리오 1. 어제 작업 이어가기

사용자:

> 어제 하던 거 이어서 해줘.

하네스:

1. Obsidian에서 마지막 세션 기록을 읽습니다.
2. 현재 프로젝트 상태를 요약합니다.
3. 미완료 결정, 오류, 작업을 보여줍니다.
4. 오늘 이어갈 수 있는 선택지를 제안합니다.
5. 사용자의 선택을 받은 뒤 다음 단계로 진행합니다.

### 시나리오 2. 새 기능 추가

사용자:

> 로그인 기능 붙여줘.

하네스:

1. 바로 구현하지 않습니다.
2. 로그인 기능에 숨어 있는 결정을 드러냅니다.
3. MVP에서 로그인 없이 가능한지 설명합니다.
4. 선택지를 제시합니다.
5. 사용자가 결정하면 Decision Record를 저장합니다.
6. 승인된 범위만 계획하고 구현합니다.

### 시나리오 3. 오류 해결

사용자:

> 이 에러가 왜 나는지 모르겠어.

하네스:

1. 오류 로그를 읽습니다.
2. Obsidian의 기존 Error Record와 Playbook을 검색합니다.
3. 반복 오류인지 확인합니다.
4. 원인 후보와 확인 순서를 쉬운 말로 설명합니다.
5. 수정 전 사용자가 이해해야 할 영향이 있으면 설명합니다.
6. 수정 후 검증 결과를 저장합니다.

### 시나리오 4. 완료 여부 확인

사용자:

> 다 된 거야?

하네스:

1. Completion Evidence Board를 확인합니다.
2. 체크된 증거와 부족한 증거를 나눕니다.
3. “구현 완료”, “로컬 기준 완료”, “검증 전 부분 완료”처럼 정확한 상태로 답합니다.
4. 다음 확인 작업을 제안합니다.

## 배포 전략

Senpai Harness의 1차 배포 형태는 **Claude Code Plugin**입니다.

MCP는 1차 배포 수단이 아니라, 2단계 확장에서 Obsidian 검색, 세션 메모리 조회, Agent Graph 분석, Edge Log 질의 같은 고급 기능을 연결하기 위한 선택 확장 계층으로 둡니다.

### 1차 배포: Claude Code Plugin

1차 MVP는 사용자가 GitHub 저장소를 Claude Code plugin marketplace로 추가한 뒤 바로 설치할 수 있는 구조를 목표로 합니다.

배포 단위:

- agents
- skills
- commands
- hooks
- Obsidian vault template
- templates
- install / doctor scripts
- plugin metadata

사용자 설치 흐름:

```text
/plugin marketplace add <github-owner>/<repo>
/plugin install senpai-harness@senpai-harness
/senpai-harness:init
```

### 2단계 확장: Optional MCP Server

MCP는 MVP 이후 선택 확장으로 추가합니다. MCP가 담당할 수 있는 기능은 다음과 같습니다.

- Obsidian Vault 검색
- 과거 세션 메모리 조회
- Error Record / Playbook 검색
- Agent Graph 조회
- Edge Log 분석
- 프로젝트 상태 질의
- 장기 기억 기반 recall

MCP는 하네스의 핵심 배포 방식이 아니라, Plugin 내부에서 선택적으로 활성화할 수 있는 고급 연결 계층입니다.

### 공개 저장소 전략

**정정 (docs 00-07/09/11 재감사)**: 원래는 구현 중인 프로젝트 저장소를 비공개 GitHub 저장소로 관리할 계획이었으나, 실제로는 원격 저장소 없이 로컬 git 저장소로만 관리되고 있다(GitHub push 이력 없음, `docs/11_DEPLOYMENT_STRATEGY.md` 참고).

하네스가 완성된 뒤에는 공개 가능한 코드와 문서를 정리하여 별도의 공개 GitHub 저장소에 Claude Code plugin marketplace 구조로 배포합니다.

공개 저장소는 사용자가 저장소를 marketplace로 추가하고 바로 설치할 수 있는 구조를 가져야 합니다.

## 성공 기준

MVP가 성공하려면 다음이 가능해야 합니다.

- 사용자가 자연어로 “이어가기”, “기능 추가”, “오류 해결”, “정리하고 종료”를 요청할 수 있습니다.
- 하네스는 명령어 없이 의도를 감지하고 적절한 회의를 엽니다.
- 새 기능 요청은 바로 구현하지 않고 숨은 결정을 먼저 보여줍니다.
- 사용자의 승인 전에는 제품 코드를 수정하지 않습니다.
- 구현 후 완료 증거가 부족하면 완료라고 말하지 않습니다.
- 작업 종료 시 Obsidian에 세션 기억이 저장됩니다.
- 반복 오류는 Error Record와 Playbook 후보로 정리됩니다.

## 비기능 요구사항

- 사용자는 명령어를 몰라도 기본 흐름을 사용할 수 있어야 합니다.
- 모든 설명은 비개발자 친화적인 쉬운 언어여야 합니다.
- 위험 작업은 반드시 사용자 승인 전 차단되어야 합니다.
- 기존 사용자 파일을 덮어쓰기 전에 백업하거나 확인해야 합니다.
- secret 파일, 환경변수, 토큰, 키 파일은 보호되어야 합니다.
- Obsidian Vault는 사람이 읽을 수 있는 Markdown 중심이어야 합니다.
