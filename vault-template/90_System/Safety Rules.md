# Safety Rules

Senpai Harness는 위험할 수 있는 작업을 사람의 판단만 믿지 않고, 코드 레벨에서 자동으로 막는 안전장치를 가지고 있습니다. 이 안전장치를 G1~G4라고 부릅니다. 이 문서는 G1~G4가 왜 이렇게 만들어졌는지 쉬운 말로 설명합니다.

이 문서는 요약이며, 정확한 원문은 `docs/SAFETY_ENFORCEMENT_POLICY.md`에 있습니다. 여기서 다르게 읽히는 부분이 있다면 원문이 항상 우선합니다.

## 무엇보다 먼저 — 비밀 파일 보호

`.env`, 비밀번호, 인증키(`id_rsa`, `*.pem`, `*.key` 같은 파일)처럼 보이는 경로는 G1~G4를 확인하기도 전에 무조건 막습니다. 읽는 것도 막습니다. 승인 상태와 상관없이 항상 막습니다. 이게 제일 먼저 체크되는 이유는, 다른 규칙이 실수로 뚫리더라도 최소한 비밀 정보만큼은 새어나가지 않게 하기 위해서입니다.

## G1 — "고칠 수도 있는 명령"은 일단 막고 본다

파일을 쓰거나(Write), 고치거나(Edit), 터미널 명령을 실행하는(Bash) 행동이 대상입니다.

- `ls`, `git status`, `git diff`, `pwd`, `grep`, `find`처럼 안전하다고 이미 알려진 읽기 전용 명령만 화이트리스트(허용 목록)로 바로 통과시킵니다.
- 화이트리스트에 없는 명령은 전부 "고칠 수도 있다"고 가정합니다. 리다이렉션(`>`, `>>`)이나 파일 내용을 바꾸는 명령, 새 프로그램을 설치하는 명령, 파일을 지우거나 옮기는 명령이 보이면 명확히 "고치는 명령"으로 분류합니다.
- **이해할 수 없는 명령은 "일단 허용"이 아니라 "일단 막습니다."** 무슨 명령인지 파악이 안 되면 무조건 막는다는 뜻입니다.
- 파일이나 폴더를 통째로 지우는 것처럼 되돌릴 수 없는 삭제 명령은, 안전한 임시 폴더가 아닌 이상 승인 여부와 상관없이 항상 막습니다.

**왜 이렇게 하는가** — 터미널 명령을 통해서도 파일을 고칠 수 있다는 게 실제로 확인됐습니다(`echo >`, `sed -i` 같은 방식). Write 버튼만 막으면 되는 게 아니라, 터미널 명령을 거쳐 몰래 파일을 고치는 길도 막아야 안전합니다.

## G2 — "한 번 허락하면 끝"이 아니라 "그때그때 다시 물어본다"

- 지금 논의 중인 범위(회의나 Decision Card에서 다뤄진 파일들) **안**에 있는 변경이면, 그때마다 사람에게 "이거 승인할까요?"라고 다시 확인합니다. 한 번 승인했다고 그다음부터 자동으로 통과되지 않습니다.
- 지금 논의 중인 범위 **밖**에 있는 변경이면, 물어보지도 않고 그냥 막고 Scope Meeting(범위를 다시 정하는 회의)으로 안내합니다. 논의되지 않은 변경은 애초에 후보가 아니라는 뜻입니다.
- "예전에 허락했던 것을 기억해두는" 별도의 장치는 일부러 만들지 않습니다. Claude Code 자체에 이미 비슷한 기능이 있어서, 굳이 하네스가 따로 만들면 오히려 더 헷갈리고 복잡해지기 때문입니다.

**왜 이렇게 하는가** — "명령어를 외우지 않아도 된다"는 편의성과 "중요한 변경은 매번 사람이 확인해야 한다"는 안전성은 서로 부딪힙니다. 이 규칙은 그 둘을 "범위 안이면 매번 물어본다, 범위 밖이면 아예 막는다"로 단순하게 풀어낸 것입니다.

## G3 — 주소를 정확하게 비교한다

파일 경로를 비교하기 전에 항상 표준 형태로 정리합니다. 상대 경로, `../`처럼 상위 폴더로 빠져나가려는 경로, 파일의 다른 이름(심볼릭 링크)까지 모두 실제 경로로 바꾼 다음 비교합니다.

**왜 이렇게 하는가** — 경로를 있는 그대로만 비교하면, 겉으로는 범위 밖처럼 보이지만 실제로는 같은 파일을 가리키는 경로로 규칙을 피해갈 수 있습니다. 주소를 정확히 정리한 뒤 비교해야 이런 우회를 막을 수 있습니다.

## G4 — 상태를 모르면 무조건 막는다 (fail-closed)

지금 세션이 회의나 승인을 거쳤는지 기록해두는 파일(`state.json`)이 없거나, 손상되어 읽을 수 없거나, 지금 세션과 맞지 않으면, 파일을 고치는 모든 행동을 무조건 막습니다. 이유는 "아직 회의나 승인이 끝나지 않은 상태"로 안내합니다.

**왜 이렇게 하는가** — 안전장치가 헷갈리는 상황에 놓였을 때 "일단 허용"으로 실패하면 위험한 변경이 그냥 통과될 수 있습니다. 반대로 "일단 차단"으로 실패하면 최악의 경우에도 아무 일도 일어나지 않습니다. 그래서 상태를 확신할 수 없을 때는 항상 막는 쪽을 선택합니다.

## 이 규칙들이 왜 이렇게까지 엄격한가

이 정책은 추측이 아니라 실측에 근거합니다. 실제 테스트에서, 안전장치가 파일 쓰기를 정확히 거부했는데도 AI 모델은 스스로 "완료했습니다"라고 잘못 보고한 사례가 두 번 확인됐습니다 (상세는 `docs/SAFETY_ENFORCEMENT_POLICY.md`의 P1 실증 검증 참고). 즉 **AI가 "괜찮다"고 말하는 것과 실제로 안전한 것은 다릅니다.** 그래서 승인과 차단은 AI의 판단이 아니라, 코드가 결정론적으로 확인하는 방식으로 만들어야 합니다. ([[Evidence Rules]] 참고)
