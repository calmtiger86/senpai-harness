# Minimality Ladder

Minimality Ladder는 과잉 구현을 막기 위한 사다리입니다. Minimality Guardian은 이 7단계를 순서대로 거쳐, 정말 필요한 만큼만, 정말 필요한 시점에만 구현하도록 합니다.

각 단계는 앞 단계에서 "그렇다"는 답이 나오지 않으면 다음 단계로 넘어가지 않습니다.

## 7단계

1. **이 기능이 지금 필요한가?**
2. **사용자가 이 기능의 목적을 이해했는가?**
3. **기존 코드나 노트에 이미 해결책이 있는가?**
4. **플랫폼 기본 기능으로 가능한가?**
5. **이미 설치된 도구로 가능한가?**
6. **더 작은 버전으로 먼저 검증할 수 있는가?**
7. **그때만 최소 구현한다.**

7단계까지 왔다는 것은 앞의 여섯 질문을 다 거쳤는데도 새로 만들어야 한다는 뜻입니다. 그 경우에만, 가장 작은 형태로 구현합니다.

## 절대 줄이면 안 되는 것

Minimality Ladder는 "덜 만들기"를 위한 것이지, "안전을 덜 지키기"를 위한 것이 아닙니다. 아래 항목은 최소 구현을 이유로 생략하거나 줄일 수 없습니다.

- 보안
- 개인정보
- 데이터 손실 방지
- 접근성
- 인증 경계
- 결제 안전성
- 사용자가 명시적으로 승인한 핵심 요구사항

참고로 실행 단계에서는 아래 작업들도 항상 별도 승인이 필요합니다 (`senpai.config.yaml`의 `safety.require_approval_for` 기준).

- 의존성 설치
- 인증 변경
- 결제 변경
- 배포 변경
- 데이터베이스 마이그레이션
- 파괴적 파일 작업

## 산출물

Minimality Guardian은 이 사다리를 거친 뒤 다음을 만들어 냅니다.

- **Minimality Check** — 몇 단계에서 멈췄는지, 왜 멈췄는지
- **simpler_path** — 더 간단한 대안이 있었다면 그 내용
- **tradeoff** — 간단한 방법과 새로 만드는 방법 사이의 트레이드오프
- **recommendation** — 어떤 방향을 추천하는지
